## 二、技術可行性評估

### 2.1 VARK 問卷 ✅ 可行

| 評估項 | 結論 | 說明 |
|--------|------|------|
| 12 題標準化 | ✅ | VARK 官方問卷約 13-16 題，12 題簡化版可行 |
| 四維度評分 | ✅ | 線性加權計算，技術簡單 |
| 行為分析追蹤 | ⚠️ | 需前端埋點支持，確認前端資源 |

**建議**: 12 題可能略少，建議參考 VARK 官方 16 題版本，或明確說明簡化依據。

### 2.2 IRT 三參數模型 (3PL) ⚠️ 需簡化

| 評估項 | 結論 | 說明 |
|--------|------|------|
| 3PL 模型實現 | ⚠️ 高難度 | 參數估計需 MLE/MCMC，計算複雜 |
| 實時調整 | ⚠️ 性能風險 | 每次作答後重新估計 θ 耗時 |
| 題目參數標定 | ❌ 缺失 | 計劃未說明如何獲得題目 a/b/c 參數 |

**重大問題**: 
- 3PL 模型需要大量作答數據才能準確估計題目參數（每題至少 200-500 次作答）
- 冷啟動階段無法使用 3PL，需降級到 1PL 或 2PL
- 實時 θ 估計建議使用貝葉斯方法（MAP 估計）而非 MLE

**建議**: 
1. **降級為 2PL 模型**（僅 a/b 參數，固定 c=0.2），降低實現難度
2. **預先標定題目參數**：使用專家評估 + 小樣本測試獲得初始 a/b 值
3. **明確冷啟動策略**：前 10 題使用固定難度序列，之後啟用 IRT

### 2.3 遺忘曲線 (Ebbinghaus + SM-2) ✅ 可行

| 評估項 | 結論 | 說明 |
|--------|------|------|
| Ebbinghaus 模型 | ✅ | 公式簡單 R = e^(-t/S) |
| SM-2 算法 | ✅ | 開源實現豐富（Anki 等） |
| 個體校準 | ⚠️ | 需至少 3-5 次復習記錄才能校準 S 值 |

**建議**: 明確冷啟動默認 S 值（建議 S=7 天），累積 5 次復習後啟用個體校準。

### 2.4 知識圖譜依賴 ⚠️ 不完整

| 評估項 | 結論 | 說明 |
|--------|------|------|
| DAG 結構 | ✅ | 標準圖結構，networkx 庫支持 |
| 前置檢查 | ✅ | 邏輯簡單 |
| 知識點覆蓋 | ❌ | 計劃未說明知識圖譜來源 |
| Dijkstra 路徑 | ⚠️ | 需考慮多目標優化（難度 + 時間） |

**重大問題**: 
- 知識圖譜構建是**勞動密集型工作**，需領域專家手動標註依賴關係
- 計劃中「先覆蓋核心 20%」過於模糊，需明確 MVP 範圍

**建議**: 
1. 明確 Phase 2 知識圖譜範圍（例如：僅限 1 個學科的前 3 章）
2. 提供知識點模板，規範專家標註格式
3. 考慮使用現成知識圖譜（如 Khan Academy 知識地圖）

## 三、時間合理性評估

### 3.1 開發時間 (4 周) ⚠️ 緊張

| 模組 | 計劃工時 | 評估工時 | 風險 |
|------|----------|----------|------|
| 學習風格識別 | 3 天 | 5 天 | 前端埋點 + 後端分析 |
| 動態難度調整 | 4 天 | 7 天 | IRT 模型調試複雜 |
| 遺忘曲線校準 | 3 天 | 4 天 | 較可行 |
| 學習路徑自適應 | 4 天 | 7 天 | 知識圖譜構建耗時 |
| API 集成 + 前端 | 3 天 | 5 天 | 依賴前端資源 |
| 數據庫 + 性能優化 | 4 天 | 4 天 | 較可行 |
| **總計** | **21 天** | **32 天** | **超支 52%** |

**結論**: 4 周開發時間**嚴重不足**，建議：
1. **延長至 6 周**，或
2. **分階段交付**：Phase 2a (學習風格 + 遺忘曲線) + Phase 2b (難度調整 + 學習路徑)

### 3.2 測試時間 (1 周) ⚠️ 不足

| 測試類型 | 計劃時間 | 建議時間 | 說明 |
|----------|----------|----------|------|
| 單元測試 | 併入開發 | 併入開發 | ✅ |
| 集成測試 | 2 天 | 3 天 | 端到端流程複雜 |
| 用戶測試 | 2 周 | 2 周 | ⚠️ 計劃寫 1 周但用戶測試需 2 周 |

**矛盾點**: 時間表中 W5 為「集成測試 + 用戶測試」共 4 天，但用戶測試明確寫「為期 2 周」。

**建議**: 用戶測試與開發並行（灰度測試），或延長總工期至 7 周。

### 3.3 分階段交付建議 ✅ 推薦

```
Phase 2a (Week 1-3): 
  - 學習風格識別（P0）
  - 遺忘曲線校準（P1）
  - 基礎 API + 前端對接

Phase 2b (Week 4-6):
  - 動態難度調整（P0，簡化版 2PL）
  - 學習路徑自適應（P1，有限知識圖譜）
  - 完整集成測試

Phase 2c (Week 7):
  - 用戶測試 + 迭代優化
  - 正式上線
```

## 四、驗收標準評估

### 4.1 學習風格識別準確率 ≥ 75% ⚠️ 需明確定義

| 問題 | 建議 |
|------|------|
| 基準是什麼？ | 與用戶自評對比？與教師評估對比？ |
| 如何測量？ | 測試後問卷：「你認為系統對你的風格判斷準確嗎？」1-5 分 |
| 75% 是準確率還是 F1？ | 建議使用 F1-score，考慮類別不平衡 |

**修改建議**: 
```
驗收標準：學習風格識別 F1-score ≥ 0.75
測量方法：30 名用戶完成問卷 + 自評風格，計算宏觀 F1
```

### 4.2 動態難度調整準確率 ≥ 80% ❌ 無法測量

**問題**: 「準確率」定義模糊——難度調整沒有標準答案。

**建議修改為可測量指標**:
```
驗收標準：
1. 用戶難度匹配度 ≥ 80%（用戶反饋「難度合適」的比例）
2. 難度跳躍平穩度：相鄰題目難度差 |Δb| ≤ 0.5 的比例 ≥ 90%
3. 挫敗感控制：連續錯誤≥3 次的場景 ≤ 5%
```

### 4.3 記憶保留率提升 ≥ 30% ⚠️ 測試週期過長

**問題**: 
- 記憶保留率需**長期追蹤**（至少 2-4 周）
- 需要對照組（使用系統 vs 不使用）
- 1 周用戶測試無法測量此指標

**建議**: 
```
驗收標準（Phase 2）:
1. 復習提醒準時率 ≥ 95%（可測量）
2. 用戶復習完成率 ≥ 70%（可測量）

驗收標準（Phase 3 追蹤）:
3. 記憶保留率提升 ≥ 30%（需 4 週對照實驗）
```

### 4.4 學習效率提升 ≥ 20% ⚠️ 需明確基準線

**問題**: 
- 基準線是什麼？（使用前測成績？平均學習時長？）
- 如何隔離其他變量？（用戶自然進步 vs 系統效果）

**建議修改為**:
```
驗收標準：
1. 單位時間知識點掌握數提升 ≥ 20%（後測/前測）
2. 用戶主觀效率評分 ≥ 4/5
測量方法：30 名用戶，2 周測試，前後測對比
```

## 五、風險評估

### 5.1 技術風險

| 風險 | 等級 | 說明 | 緩解措施 |
|------|------|------|----------|
| IRT 3PL 實現複雜 | 🔴 高 | 參數估計算法複雜，冷啟動無數據 | **降級為 2PL**，預先標定題目參數 |
| 知識圖譜構建耗時 | 🔴 高 | 需專家手動標註，Phase 2 可能無法完成 | **限制範圍**：僅 1 學科前 3 章 |
| 實時性能不足 | 🟡 中 | IRT 估計 + 路徑規劃可能超 100ms | 使用 Redis 緩存 θ 估計，異步計算路徑 |
| 前端埋點缺失 | 🟡 中 | 行為分析依賴前端數據收集 | 確認前端資源，提供埋點 SDK |

### 5.2 數據風險

| 風險 | 等級 | 說明 | 緩解措施 |
|------|------|------|----------|
| 冷啟動數據不足 | 🟡 中 | 新用戶無行為數據 | 問卷結果作為初始值，10 題後過渡 |
| 題目參數標定缺失 | 🔴 高 | 無歷史數據無法估計 a/b/c | **專家評估 + 小樣本測試**獲得初始值 |
| 用戶樣本量不足 | 🟡 中 | 30 名用戶統計效力有限 | 明確為 MVP 測試，正式上線需更大樣本 |

### 5.3 性能風險

| 風險 | 等級 | 說明 | 緩解措施 |
|------|------|------|----------|
| API 延遲超標 | 🟡 中 | 1000 QPS 下<100ms 有挑戰 | 壓力測試，必要時降級（簡化算法） |
| 數據庫瓶頸 | 🟢 低 | PostgreSQL + Redis 架構合理 | 監控慢查詢，添加索引 |

### 5.4 用戶體驗風險

| 風險 | 等級 | 說明 | 緩解措施 |
|------|------|------|----------|
| 難度跳躍過大 | 🟡 中 | IRT 估計不穩可能導致難度劇變 | 限制單次調整幅度 |Δb| ≤ 0.5 |
| 推薦內容單一 | 🟡 中 | 過度依賴主導風格 | 混合推薦：主導風格 70% + 其他 30% |
| 復習提醒騷擾 | 🟢 低 | 推送過多可能引起反感 | 允許用戶自定義提醒頻率 |

## 六、審議結果

### 審議結論：**選項 B: 准奏 + 修改意見** ⚠️

計劃基本可行，但存在以下**必須修改**的問題：

### 必須修改項（Blocking）

1. **IRT 模型降級**: 3PL → 2PL，明確冷啟動策略（前 10 題固定難度）
2. **知識圖譜範圍**: 明確 Phase 2 覆蓋範圍（建議：單學科前 3 章）
3. **時間表調整**: 4 周 → 6 周，或分階段交付（2a/2b/2c）
4. **驗收標準修訂**: 
   - 難度調整「準確率」改為可測量指標
   - 記憶保留率移至 Phase 3 驗收
5. **題目參數標定方案**: 補充如何獲得初始 a/b 值（專家評估 + 小樣本測試）

### 建議修改項（Recommended）

1. VARK 問卷增至 16 題（參考官方版本）
2. 用戶測試與開發並行（灰度測試）
3. 添加 A/B 測試框架，為 Phase 3 做準備

### 修改後行動

1. 中書省根據上述意見修改計劃（v1.1）
2. 門下省快速覆審（24 小時內）
3. 通過後轉尚書省執行
